### 2.3 Outstanding Security Issue (Not Fixed — Requires Backend)

> **⚠️ `VITE_RESEND_API_KEY` is exposed in the browser.**
>
> The `.env` file contains `VITE_RESEND_API_KEY=re_PsTXNr4y_...`. Any env var prefixed with `VITE_` is bundled into the frontend JavaScript by Vite. This means anyone can open DevTools → Network and see the API key in fetch request headers.
>
> **Risk:** Unlimited email sending from your Resend account.
>
> **Proper Fix:** Create a Supabase Edge Function (or any backend endpoint) that receives a "send email" request from the frontend (without API keys), and the backend calls Resend with the key stored securely as a server-side environment variable. The frontend should never know the Resend API key.
>
> This was **not** changed in this session because `notifications.ts` (complaint notifications) also uses this key from the frontend, and restructuring that requires a backend setup that was out of scope.

## Phase 3: Logic Bug Fixes

### 3.1 AuthContext `useEffect` Dependency Array Bug

|             | Before                                                                                                                                                                 | After                        |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| **File**    | `src/context/AuthContext.tsx` line 128                                                                                                                                 | Same file, same location     |
| **Code**    | `useEffect(..., [user, profile])`                                                                                                                                      | `useEffect(..., [user?.id])` |
| **Problem** | The effect fetches the profile and calls `setProfile(data)`. Since `profile` is in the dependency array, setting it triggers the effect again. The guard `if (!profile |                              | profile.id !== user.id)` prevents an infinite loop in the happy path, but edge cases (e.g., profile fetch returns different data shapes, or errors) could cause extra re-renders. | Now only re-runs when `user?.id` actually changes (new login or logout). |

### 3.2 Signup Race Condition (Fixed via Rewrite in Phase 4)

**Before:** `SignupPage.tsx` called `supabase.auth.signUp()` and then immediately tried to `upsert` into `profiles` and `citizens` tables. Since the user's email is unconfirmed at this point, Supabase RLS blocks the insert. The code caught the error, logged "Profile creation deferred until after email verification," and silently continued. Then `VerifyPage.tsx` did the exact same upsert after OTP verification — meaning the same upsert logic was duplicated in 2 files.

**After:** Profile and citizen creation is done **only** in `VerifyPage.tsx` after OTP verification succeeds. `SignupPage.tsx` only handles auth signup + OTP generation/sending.

### 3.3 VerifyPage Missing Email Guard (Fixed via Rewrite in Phase 4)

**Before:** If a user navigated directly to `/verify` (e.g., bookmarked it), `location.state` is null, so `email` is empty. The fallback `getUser()` only works if there's already a session. If neither works, the form submits with an empty email, causing a confusing Supabase error.

**After:** When no email is available from navigation state, the page shows a text input where the user can manually type their email.

## Phase 4: OTP Migration from Supabase to Resend

### Why This Migration Was Needed

From the project's conversation history:

- Supabase free tier limits email sending to **3 emails per hour**
- Users were repeatedly not receiving verification emails
- Multiple debugging sessions (conversations `99034ebc`, `aa9abcb2`, `09328e39`) were spent troubleshooting this

### What Changed

The entire OTP flow was replaced: Supabase's built-in email OTP system → Custom OTP stored in Supabase DB + Sent via Resend API.

### New File: `src/lib/otp.ts`

A new utility module with 4 exported functions:

| Function                               | Purpose                                  | Details                                                                                                                                                                               |
| -------------------------------------- | ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `generateOTP()`                        | Creates a 6-digit code                   | Uses `window.crypto.getRandomValues()` for cryptographic randomness. Falls back to `Math.random()` if unavailable. Always produces a 6-digit number (100000-999999).                  |
| `storeOTP(email, code)`                | Saves code to Supabase `otp_codes` table | First invalidates any previous unused codes for the same email (marks them `used: true`). Then inserts a new row with a 10-minute expiry.                                             |
| `verifyOTP(email, code)`               | Checks code against stored value         | Queries `otp_codes` for a matching email + code that is unused and not expired. If found, marks it as used and returns `{ valid: true }`.                                             |
| `sendOTPEmail(email, code, userName?)` | Sends OTP via Resend API                 | Uses `fetch()` to call `https://api.resend.com/emails` with a styled HTML email template containing the 6-digit code. Sends from `onboarding@resend.dev` (Resend's free-tier sender). |

### New File: `migrations/otp_codes_table.sql`

SQL migration to create the `otp_codes` table. See the "SQL Migration Guide" section below for full details.

### Modified File: `src/app/pages/auth/SignupPage.tsx`

**Complete rewrite.** Key changes:

- Removed `const [role] = useState<"citizen">("citizen")` (unnecessary state, just use string literal)
- Removed redundant `upsert` calls to `profiles` and `citizens` tables
- Added call to `generateOTP()` → `storeOTP()` → `sendOTPEmail()` after `supabase.auth.signUp()`
- Passes `userId` in navigation state to verify page
- Changed input type to `email`, added `minLength={6}` to password input
- Added client-side password length validation before making any API calls

### Modified File: `src/app/pages/auth/VerifyPage.tsx`

**Complete rewrite.** Key changes:

- Replaced `supabase.auth.verifyOtp()` with custom `verifyOTP()` from `otp.ts`
- Replaced `supabase.auth.resend()` with custom `generateOTP()` + `storeOTP()` + `sendOTPEmail()`
- Added manual email input field when email is not in navigation state
- Added 60-second cooldown timer on "Resend Code" button to prevent abuse
- Profile/citizen creation now happens here (post-verification only)
- Added proper fallback: if no session exists after verification, redirect to login with success message

## SQL Migration Guide

### What the Migration Creates

The file `migrations/otp_codes_table.sql` creates the following in your Supabase database:

**1. Table: `otp_codes`**

| Column       | Type        | Default             | Description                                          |
| ------------ | ----------- | ------------------- | ---------------------------------------------------- |
| `id`         | UUID        | `gen_random_uuid()` | Primary key                                          |
| `email`      | TEXT        | (required)          | The user's email, stored lowercase                   |
| `code`       | TEXT        | (required)          | The 6-digit OTP code                                 |
| `expires_at` | TIMESTAMPTZ | (required)          | When the code expires (set to 10 min after creation) |
| `used`       | BOOLEAN     | `false`             | Whether the code has been used/invalidated           |
| `created_at` | TIMESTAMPTZ | `NOW()`             | When the code was created                            |

**2. Indexes:**

- `idx_otp_codes_email` — For fast lookups when verifying codes
- `idx_otp_codes_expires_at` — For cleanup queries

**3. Row Level Security (RLS) Policies:**

- `Allow insert for signup` — Allows anyone to insert (needed because users aren't authenticated during signup)
- `Allow read own codes` — Allows anyone to read (needed for verification before auth)
- `Allow update codes` — Allows anyone to update (needed to mark codes as used)

> **Note:** These policies are permissive because the OTP system operates _before_ the user is authenticated. In production, you should move OTP handling to a Supabase Edge Function where the database operations happen server-side.

**4. Cleanup Function:** `cleanup_expired_otps()` — Deletes codes expired for more than 1 hour. Can be set up as a Supabase cron job for automatic cleanup.

### How to Run

1. Open your **Supabase Dashboard** → **SQL Editor**
2. Paste the **entire** contents of `migrations/otp_codes_table.sql`
3. Click **Run**
4. You should see "Success. No rows returned" — this is normal for DDL statements

### What to Expect After Running

- A new `otp_codes` table will appear under **Table Editor** in Supabase
- RLS will show as "Enabled" with 3 policies
- The table will be empty until someone signs up

## Phase 5: Post-Testing Fixes (Evening Session)

### 5.1 CORS Fix — All Resend API Calls Moved Server-Side

**Problem:** The Resend API (`https://api.resend.com/emails`) blocks browser-to-API calls via CORS policy. This affected ALL email sending: OTP verification, complaint confirmations, and staff assignment notifications.

**Fix:** Created a generic `send_email` PostgreSQL function using Supabase's built-in `pg_net` extension. This makes HTTP calls from the database server, completely bypassing CORS. All three email types now route through this single server-side function.

| File                                 | Before                                          | After                                        |
| ------------------------------------ | ----------------------------------------------- | -------------------------------------------- |
| `src/lib/otp.ts`                     | Called `send_otp` RPC (OTP-specific)            | Calls `send_email` RPC (generic)             |
| `src/lib/notifications.ts`           | Direct `fetch("https://api.resend.com/emails")` | Calls `send_email` RPC                       |
| `migrations/send_email_function.sql` | —                                               | **New** — Generic server-side email function |

### 5.2 Profile Auto-Creation on Login

**Problem:** When a user's session expired between signup and OTP verification, the profile/citizens entries were never created. LoginPage then threw "Account integrity error: Profile missing."

**Fix:** Three layers of protection added:

- **LoginPage:** Auto-creates profile + citizens entry if missing
- **LoginPage:** Always checks citizens entry exists for citizen accounts (prevents FK errors when filing complaints)
- **AuthContext:** Auto-creates profile if missing instead of force signing out
- **VerifyPage:** Attempts profile creation even when session is expired (using userId from navigation state)

### 5.3 Citizens Table FK Fix

**Problem:** Even after profile was fixed, filing a complaint threw `"violates foreign key constraint 'raw_complaints_user_id_fkey'"` because `raw_complaints.user_id` references `citizens(id)` and the citizens entry didn't exist.

**Fix:** LoginPage now checks for the citizens entry on **every citizen login** and creates it if missing, before the user reaches the dashboard.

### 5.4 Improved Role Mismatch Error Message

**Before:** `"Unauthorized: This account is registered as a admin, not citizen."`
**After:** `"This account is registered as a admin. Please select "admin" above and try again."`
